Digital Signature vs MAC

cryptography cheat sheet

Digital Signature
🔏
Asymmetric (Public/Private Key)
Sender signs with their private key. Anyone with the public key can verify. Provides non-repudiation: the signer cannot deny authorship.
MAC
🤝
Symmetric (Shared Secret Key)
Both parties share the same secret key. Either side can compute and verify the tag. No non-repudiation: both could have created it.

Secure Printing Technologies

💧
Watermarks
Faint images or patterns embedded into paper during manufacturing, visible when held to light. Used in banknotes and official documents.
🌈
Holograms
Metallic labels that shift in color and pattern at different viewing angles. Found on credit cards, passports, and product packaging.
🔍
Microprinting
Extremely small text unreadable to the naked eye. Photocopying or scanning destroys legibility, making forgery detectable. Used on checks and currency.
🔮
UV / Fluorescent Ink
Ink invisible under normal light but glows under ultraviolet light. Used on event tickets, stamps, and banknotes for hidden verification.
🧵
Security Threads
Thin metallic or plastic strips embedded within paper. May contain microtext or color-shifting properties. Nearly impossible to replicate with standard printers.
🚫
Tamper-Evident Printing
Special inks or coatings that reveal visible damage (e.g. "VOID" patterns) when altered or peeled. Common on prescription pads and secure labels.
🌀
Guilloché
Intricate, repetitive spirograph-like line patterns engraved with high precision. Extremely difficult to reproduce accurately. Found on banknotes, certificates, and diplomas.

Vernam Cipher (One-Time Pad)

theoretically unbreakable, practically challenging

How It Works
01 Sender and receiver share an identical truly random key that is at least as long as the plaintext message.
02 Each character (or bit) of the plaintext is combined with the corresponding key character using XOR (binary) or modular addition (letters).
03 To decrypt, the receiver applies the same operation in reverse using the identical key.
Example (mod 26)
Plaintext: H (7)  +  Key: D (3)  =  (7+3) mod 26 = 10  →  Cipher: K
Ciphertext: K (10)  −  Key: D (3)  =  (10−3) mod 26 = 7  →  Plain: H
Why It Is Theoretically Perfect
If the key is truly random, used only once, and kept secret, the ciphertext reveals zero information about the plaintext. Every possible plaintext is equally likely for any given ciphertext. This is called perfect secrecy, proven mathematically by Claude Shannon.
Practical Limitations
Key Length
The key must be exactly as long as the message, making generation and storage a major challenge for large data volumes.
Key Distribution
The key must be securely delivered to both parties beforehand. If a secure channel exists for the key, you could arguably just send the message through it.
No Key Reuse
Reusing any part of a key destroys security. An attacker can XOR two ciphertexts to eliminate the key and reveal plaintext patterns.
True Randomness
Generating truly random keys (not pseudorandom) at scale is difficult and resource-intensive.

CPS Robustness Criteria

five properties of dependable cyber-physical systems

⏱️
Reliability
R(t)
The probability that the system is working correctly at time t, given that it was working correctly at t = 0. Measures how likely the system is to keep running without failure from the start.
🔧
Maintainability
M(d)
The probability that the system is working correctly again after duration d has passed since an error occurred. Captures how quickly and likely the system is to recover after a fault.
✅
Availability
A(t)
The probability that the system is working correctly at any given time t, regardless of what happened before. Unlike reliability, it does not assume the system was working at t = 0.
🔒
Security
Ensures confidential and authenticated communication, control, and storage across the system.
🛡️
Safety
The system causes no harm to people, property, or the environment, even when faults occur.

Microprocessor Security Attacks

two hardware-level exploitation techniques

Plundervolt
Exploits: Voltage Scaling
01 Attacker (with root privileges) uses the CPU's software-controlled voltage interface to deliberately underpower the processor.
02 The voltage drop is timed during a sensitive computation inside a secure enclave (e.g. Intel SGX), causing bit flips or computation errors.
03 The attacker analyzes the faulty outputs to extract secret keys or bypass the enclave's integrity checks.
Meltdown
Exploits: Out-of-Order Execution
01 The CPU speculatively reads kernel memory that the user process should not have access to.
02 The CPU eventually checks permissions and blocks the access, but the data has already been loaded into the CPU cache.
03 The attacker uses a cache timing side-channel to determine which data was loaded, effectively reading protected kernel memory from an unprivileged process.

Buffer Overflow Attack

01 A program allocates a fixed-size buffer in memory but does not check the size of user input before writing to it.
02 The attacker sends input larger than the buffer, causing excess data to overwrite adjacent memory, including the function's return address on the stack.
03 The return address is replaced with a pointer to the attacker's injected malicious code (shellcode).
04 When the function finishes, it jumps to the overwritten return address and executes the attacker's code instead of returning normally.

Public Key Infrastructure (PKI)

entities that enable trust in public-key systems

🏛️
Certificate Authority (CA)
Trusted entity that issues, signs, and revokes digital certificates. Verifies identity before binding a public key to it.
📋
Registration Authority (RA)
Intermediary between users and the CA. Handles identity verification and certificate requests but does not issue certificates itself.
🗄️
Certificate Repository
Public directory or database storing issued certificates and CRLs, accessible for anyone to retrieve and verify.
👤
End Entity
The user, device, or service that holds a certificate. The party whose identity is bound to a public key.
✅
Validation Authority (VA)
Provides real-time certificate status checking (e.g. via OCSP) so relying parties can confirm if a certificate is still valid or revoked.

Hard vs Soft Real-Time Constraints

Hard Real-Time
Deadline must be met every single time. Missing it causes total system failure or catastrophic consequences.
An airbag controller must deploy within milliseconds of a collision. If the deadline is missed, the airbag is useless.
Security Implication
Attacker floods the airbag sensor network with fake data (DoS). The ECU is too busy processing junk to respond within the hard deadline during a real collision, causing the airbag to fail to deploy.
Soft Real-Time
Deadline should be met, but occasional misses are tolerable. Results in degraded performance, not failure.
A video stream should render at 30fps. A few dropped frames cause minor stuttering but the system continues functioning.
Security Implication
Attacker causes resource exhaustion on a security camera's feed processing. Footage becomes laggy or frames drop, potentially missing critical moments during a break-in. System works, but effectiveness is reduced.

Hardware Security Modules (HSM)

🌐
Network Attached HSM
Advantage: Shared access across many servers and applications, scalable for enterprise use.
Disadvantage: Introduces network latency and is exposed to network-based attacks.
🔌
PCIe Embedded HSM
Advantage: Fastest performance due to direct hardware bus communication with minimal latency.
Disadvantage: Tied to a single server, scaling requires additional units per machine.
🔑
USB / Portable HSM
Advantage: Highly portable and affordable, easy to move between machines for tasks like code signing.
Disadvantage: Limited processing power and throughput, not suited for high-volume enterprise workloads.
☁️
Cloud HSM
Advantage: No physical hardware to maintain, easily scalable on demand, accessible from anywhere.
Disadvantage: Keys managed in a third-party environment, may not meet strict compliance requirements, relies on internet connectivity.

WEP vs WPA/WPA2

WEP
Encryption
RC4 stream cipher with a static 40-bit or 104-bit key. A 24-bit IV is prepended per packet, but the small IV space causes repeats quickly, allowing attackers to crack the key.
Key Distribution
The same static key is shared across all devices on the network and rarely changes.
WPA / WPA2
Encryption
WPA uses TKIP (still RC4 but with unique per-packet keys and a 48-bit IV). WPA2 upgraded to AES-CCMP, a much stronger block cipher.
Key Distribution
A 4-way handshake derives unique session keys (PTK) from a Pre-Shared Key (PSK) or via 802.1X/RADIUS for enterprise. No device shares the same encryption key.
WEP: static shared key + weak encryption = easily cracked. WPA/WPA2: dynamic per-session keys + stronger encryption = significantly more secure.

Timing Attack

How It Works
01 Cryptographic operations take different amounts of time depending on the input data.
02 The attacker repeatedly sends different inputs and precisely measures response times.
03 By analyzing time variations, the attacker deduces secret information like encryption keys or passwords.

Digital Signature Protocol

01
Key Generation
The signer generates a public/private key pair. The private key is kept secret, the public key is shared.
02
Hashing
The original message is passed through a hash function (e.g. SHA-256) to produce a fixed-length digest. Signing a small hash is faster than signing the entire message.
03
Signing
The signer encrypts the hash using their private key. This encrypted hash is the digital signature, attached to the original message.
04
Transmission
The original message + digital signature are sent together to the receiver.
05
Verification
The receiver decrypts the signature using the signer's public key to recover the hash. The receiver also independently hashes the received message. If both hashes match, the signature is valid.

MAC + Encryption Modes

Most Secure
Encrypt-then-MAC (EtM)
Plaintext → Encrypt → Ciphertext → MAC
Plaintext is encrypted first, then a MAC is computed over the ciphertext. Receiver verifies MAC before decrypting. Tampered ciphertext is rejected before decryption even occurs.
✓ Recommended approach
Weaker
MAC-then-Encrypt (MtE)
Plaintext → MAC → Encrypt(Plaintext + MAC)
MAC is computed over plaintext first, then both plaintext and MAC are encrypted together. Receiver must decrypt before verifying the MAC, processing potentially tampered ciphertext.
⚠ Must decrypt before verifying
Weakest
Encrypt-and-MAC (E&M)
Plaintext → Encrypt & MAC(Plaintext) independently
Plaintext is encrypted and a MAC is computed over the plaintext independently. Both sent together. The MAC may leak information about the original message since it is computed on unencrypted data.
✗ MAC may leak plaintext info

Keyless Car Entry Attack

Relay Attack
01 Attacker A stands near the car with a relay device. Attacker B stands near the owner (key fob in pocket or home).
02 The relay devices amplify and forward the key fob's signal to the car, tricking it into thinking the fob is nearby.
03 The car unlocks and engine starts even though the actual key is far away.

Remedy
Use a distance-bounding protocol that measures the round-trip time of the signal between car and key fob. Relay attacks add latency from the extra forwarding step, so the car detects the response took too long and rejects the unlock request. A simpler measure is storing the key fob in a Faraday pouch to block signals when not in use.

IoT-Enabled CPS Security

security requirements across six properties

Example System
Smart Healthcare Patient Monitoring — wearable sensors track vitals (heart rate, blood pressure, oxygen) and transmit data to hospital servers via the internet for real-time monitoring by doctors.
Confidentiality
Patient health data must be encrypted during transmission and storage so only authorized medical staff can access it. A breach could expose sensitive medical records.
Integrity
Sensor readings must not be altered in transit. Tampered vital signs (e.g. modified heart rate) could lead to incorrect diagnosis or dangerous treatment decisions.
Availability
The system must be operational 24/7. Downtime or a DoS attack could mean critical changes in a patient's condition go undetected.
Authenticity
Commands to medical devices (e.g. adjusting insulin pump dosage) must be verified as coming from an authorized doctor, not a spoofed source.
Non-Repudiation
Every action (e.g. remotely adjusting medication) must be logged with proof of who issued it, so no party can deny their actions in case of malpractice.
Privacy
Must comply with regulations (e.g. HIPAA) ensuring patient data is only used for its intended purpose and not shared with unauthorized third parties without consent.

Kerckhoffs' Principle

Principle
A cryptographic system should remain secure even if everything about the system is publicly known, except for the secret key. Security must depend solely on the secrecy of the key, not on the secrecy of the algorithm.
Can Be Public
The algorithm, its design, source code, and implementation details. E.g. AES is published openly and peer-reviewed.
Must Stay Secret
Only the key. Even if an attacker knows exactly which algorithm is used and how it works, they cannot break it without the key.

Semi-Autonomous Vehicle Privacy

📍
Location Tracking
Violation
GPS and V2X communication data can be intercepted or collected by third parties, revealing the driver's daily routes, frequent destinations, and movement patterns.
Remedy
Use pseudonym certificates that periodically rotate the vehicle's identifier during V2X communication. The vehicle cycles through temporary pseudonyms so no entity can link all location data to one driver over time.
🎥
In-Cabin Data Exposure
Violation
Cameras and microphones for driver monitoring (drowsiness detection, voice commands) can be accessed or leaked, exposing private conversations and behavior.
Remedy
Use homomorphic encryption to process sensor data in encrypted form. The monitoring system analyzes encrypted feeds without decrypting raw data, so even if intercepted, the actual footage remains unreadable.

p = 5, q = 7, plaintext M = 10

Key Generation
01 Choose primes: p = 5, q = 7
02 Compute n = p × q = 35
03 Compute Euler's totient: φ(n) = (p−1)(q−1) = 4 × 6 = 24
04 Choose e such that 1 < e < 24 and gcd(e, 24) = 1. Pick e = 5
05 Compute d such that d × e ≡ 1 (mod 24). d = 5 (since 5 × 5 = 25, 25 mod 24 = 1)

Pairwise Keying vs Group Keying

Pairwise Keying
Advantage: Compromising one node only exposes keys shared with that specific node. Rest of the network remains secure (strong resilience).
Disadvantage: Each node must store n−1 keys, which is not scalable for resource-constrained sensors with limited memory.
Group Keying
Advantage: Each node stores only one key regardless of network size, making it highly memory-efficient and scalable.
Disadvantage: Compromising any single node exposes the group key, allowing the attacker to eavesdrop on all communication in that group (weak resilience).

Bluetooth Vulnerabilities

💬
Bluejacking
Attacker sends unsolicited messages to nearby devices via OBEX push. Mostly harmless but can be used for phishing or social engineering.
Prevention: Disable Bluetooth discoverability when not in use.
📂
Bluesnarfing
Attacker exploits OBEX vulnerability to gain unauthorized access to data (contacts, messages, calendars) without the owner's knowledge.
Prevention: Keep firmware updated and use strong authentication before allowing any data transfer.
🎛️
Bluebugging
Attacker takes full control of a device by exploiting flaws in the pairing process, allowing them to make calls, send messages, and access the internet.
Prevention: Use Secure Simple Pairing (SSP) with strong mutual authentication and keep devices on the latest Bluetooth version.
